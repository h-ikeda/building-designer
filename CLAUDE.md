# CLAUDE.md — セッション動作ルール

このファイルは、building-designer リポジトリで作業する AI（Claude）向けの動作ルールです。
外部の人間向けドキュメントは日英併記ですが、**このファイルは AI が読むためのものなので
日本語のみ**で記述します。

---

## 0. ミッション

日本の建築物をワンストップで設計するソフトウェア（3D モデリング、法適合性照合、
消費エネルギー計算、構造計算など）を開発する。現在は **基本設計フェーズ** であり、
複数セッションを通じて要件・ドメインモデル・アーキテクチャ方針を固めている段階。

**詳細は必ず [`README.md`](./README.md) と [`docs/index.md`](./docs/index.md) を読むこと。**
このファイルはあくまで「各セッションでどう動くか」を定めるもので、プロジェクトの
設計方針・ロードマップは README、設計文書は `docs/` にある。

---

## 1. セッション開始手順

作業を始める前に、必ず以下の順で現状を把握する。

1. [`README.md`](./README.md) を読み、ミッション・方針・ロードマップ・現在のフェーズを確認。
2. [`docs/index.md`](./docs/index.md) を読み、設計文書の索引と運用ルールを確認。
3. 直近の [`docs/adr/`](./docs/adr/) と関連する [`docs/domains/`](./docs/domains/) を読み、
   これまでの決定事項を把握。
4. **オープンな GitHub Issue を確認**（`list_issues`）し、未決の論点と検討状況を把握。

これらを踏まえ、そのセッションで扱う論点を明確にしてから作業に入る。

---

## 2. 基本設計フェーズの原則

- **技術スタックを安易に確定しない。** 機能スコープも技術も未確定であり、柔軟に
  見直す前提で検討する。特定の言語・フレームワークを決め打ちしない。
- **未決事項は GitHub Issue、決定事項は docs。** この分業を徹底する。
  - `docs/` に「未決」「TBD」「検討中」を残さない。未決は必ず Issue 側で追う。
  - 検討中・要判断の論点が生じたら Issue を起票する。
- **決定が下りたときの流れ:**
  1. [`docs/adr/`](./docs/adr/) に ADR を追加して決定を記録する。
  2. 必要に応じて [`docs/domains/`](./docs/domains/) や
     [`docs/architecture.md`](./docs/architecture.md) の該当箇所に反映する。
  3. 元の Issue に決定要約と ADR へのリンクをコメントし、`type:decision` ラベルに
     更新してクローズする。

---

## 3. GitHub の使い方

### 3-1. ラベル体系

Issue は次のラベルで分類する（未作成のラベルは起票時に付与を試み、付与できない
場合はこの一覧を基に手動作成すればよい）。

- 領域別: `domain:3d` / `domain:legal` / `domain:energy` / `domain:structural`
- 種別: `type:question`（未決の論点）/ `type:decision`（決定済み）/ `type:adr`

### 3-2. Issue テンプレート

未決事項は `.github/ISSUE_TEMPLATE/design-question.yml`（設計論点フォーム）を使い、
「論点 / 選択肢 / 判断材料 / 影響領域 / 関連ドキュメント」を埋めて起票する。粒度を
セッション間で揃えるため、原則このフォームを用いる。

### 3-3. サブイシュー

「対象機能スコープの確定」のような大きな論点は、親 Issue を立て、領域別（3D/法令/
エネルギー/構造）などの子 Issue に分解してよい（`sub_issue_write`）。

### 3-4. ADR ↔ Issue の相互リンク

- ADR 本文には、解決した Issue 番号（`解決した Issue: #N`）を記載する。
- Issue クローズ時には、対応する ADR へのリンクを残す。

## 4. ドキュメント記述ルール

- **言語:** README と `docs/` 配下は **日英併記**（外部の人間が読む可能性があるため）。
  この `CLAUDE.md` のみ日本語で書く。
- **用語の統一:** 日本の建築法令用語は正式名称で統一する（例: 建築基準法、省エネ基準、
  構造関係規定）。訳語も文書内で揺らさない。
- **決定事項ベース:** `docs/` は確定した設計を表す。推測や検討中の内容を断定的に
  書かない。

  `claude/building-design-docs-setup-lm6plx` で作業する。
- 明確で説明的なコミットメッセージを書く。
- プッシュは `git push -u origin claude/building-design-docs-setup-lm6plx`。
- プルリクエストはユーザーの明示的な指示があるまで作成しない。
